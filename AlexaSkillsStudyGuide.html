Alexa Overview

Four Design Patterns Unique to Voice

  1. Be Adaptable: Let users speak in their own words
  2. Be Personal: Individualize entire interactions
  3. Be Available: Collapse menus and Make all options top level (no nesting)
  4. Be relatable: Talk with your users - don't talk at them

Be Adaptable

  • The "OK" button is now the intent
  • The intent is another word of what the user want's to accomplish
  • In voice, we only have audio. There is no screen options or buttons to guide the user
  • Utterances are the things that users say.

      • Examples of utterances include

    • "Ok"
    • "Next"
    • "Now what"
    • "Got it"
  • Natural UI (User Interface) will accomodate all types of responses
  • Uses Automatic Speech Recognition (ASR) and Natural Language Understanding (NLU) will resolve to an "Ok" intent
  • The technology must conform to the user's style of speech
  • Developers can provide clues to users as to what to say
  • The goal of Voice User Interface (VUI)is to allow customers the free in what to say and not worry about specific wording in the user interface

Be Personal

  • Individualize the entire interaction
  • Create a personal experience
  • Create random welcomes to greet your users. Don't say the same "Hello" for every greeting.
  • First time users for your skill will need more instructions.
  • Over repeated use, users won't need to hear the long introductions or instructions
  • An experience with memory allows users to pick up where they left off
  • An experience with memory can customize the greeting for repeat users

Be Available

  • Don't use hierarchial menus (nesting) with VUI
  • When using hierarchial menus it constrains the UX (User Experience) and makes the user have to think too much on how the skill is organized

Be Relatable

  • Use dialogue management to manage the state and flow of the conversation.
  • "Talk with the users", not "Talk at the users"

Build a Skill

AWS Resources Used with Alexa Skills

  • Code editor in dev console
  • Changes to Lambda are coded in the developer console
  • Can also use AWS CLI (Command Line Interface) to edit and deploy an Alexa hosted skill

The Setup

  • Can build, edit and publish in the dev console
  • Dev console has access to
    1. Lambda endpoint
    2. S3 for media storage
    3. S3 backed key value table for managing session persistence
  • The Code Editor Allows
    1. Editting of the backend code
    2. Deploy directly into Lambda
  • When skill is create, the service sets up AWS CodeCommit repo for maanging the code (this can be accessed via AWS CLI to edit)
  • From the dev console, can create an Alexa hosted skill from the "Create Skill" page
  • From the dev console the bottom of the page shows optiosn for hosting backend sources

Types of Skills

  • Custom Skills don't use pre-built models

AWS Accounts

  • Don't need an AWS account to create or edit in Alexa developer console
  • If using other AWS services beyond the Alexa dev console - will need to create AWS account

NPM and PIP

  • AWS Codebuild is part of a skill to execute the build script
  • NPM or PIP install executes as part of build script
  • No need to run NPM install, because can put dependencies in package.json
  • AWS Codebuild runs NPM install when click the "Deploy" button

Confirm

  • Hosted skills periodically sends available check to Lambda functions
  • This helps with ensuring that calls to AWS Lambda from Alexa are good
  • Done in form of a LaunchRequest

Usage Limits

    Limited to AWS Free Tier
  • Lambda: 1 million free AWS Lambda requests per month
  • Lambda: 3.2 millions seconds of computer time per month
  • S3: 5GB of S3 storage per month
  • S3: 20,000 GET requests per month
  • S3: 2,000 PUT requests per month
  • S3: 15GB data transfer out per month
  • CodeCommit: 50GB per month storage
  • CodeCommit: 10,000 GET requests per month
